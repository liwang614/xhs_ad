# 2026-02-19 Commit Memory

这次改动围绕“AI判断帖子是否为求助帖并写入数据库标记”落地，同时补齐了配置化切换和多表处理能力，并把执行说明沉淀到 memory 目录，确保后续 AI 或开发者接手时可以直接复用。

在新增能力上，项目新增了独立判定模块 `modules/help_post_judge.py`，把模型调用从业务流程里拆出，统一支持 openai、gemini、claude 三个提供方。该模块实现了一致的输入提示词、统一 JSON 解析策略、失败兜底处理和日志落盘机制。API Key 的读取也被标准化为优先读取 `LLM_API_KEY`，缺失时再按 provider 读取专属环境变量，这样既兼容多环境，也避免把密钥写入仓库。与此同时，数据库层新增了 `is_help_post` 的自动补列逻辑和按表维度的判定读写接口，保证判定结果可以稳定落到目标表。

在功能变更上，`modules/config_loader.py` 的 `ai_judge` 配置被扩展为可描述完整判定策略，除了已有的 mode、provider、model、batch_size、timeout、base_url，还支持 `tables`。这个字段既可以是单个字符串，也可以是字符串数组，意味着可以在同一套参数下完成单表或多表判定。`tools/classify_help_posts.py` 已改为按 `ai_judge.tables` 顺序逐表执行，处理 `is_help_post IS NULL` 的记录并输出每个表的成功失败统计。`main.py` 的 batch 判定入口也同步支持多表轮询，避免手工改环境变量反复运行。

这批改动主要解决了三个现实问题。第一个问题是判定逻辑分散，模型调用、解析、异常处理和日志记录耦合在脚本中，不利于稳定维护；现在通过独立模块把调用协议和错误策略统一，后续扩 provider 或换模型只需改一处。第二个问题是目标表切换成本高，过去依赖环境变量或改代码，容易误操作；现在通过 `config.json` 直接声明 `ai_judge.tables`，单表与多表切换都可配置化完成。第三个问题是失败可观测性不足，模型返回异常格式或接口失败时不容易追踪；现在失败会保留数据库 `NULL` 状态并写入 `logs/help_post_judge_errors.log`，日志包含时间、row_id、provider、model 与错误原因，便于回溯和重跑。

实现思路上，这次优先选择“最小侵入、边界清晰”的策略。判定相关变化尽量集中在配置层、数据库层与判定脚本，评论发送和文案逻辑不在这部分内扩展行为。数据库写入保持幂等取向：只有拿到有效的 `0/1` 判定才更新字段，任何失败都不覆盖原值。表名输入增加了格式校验，只允许字母、数字和下划线，减少动态 SQL 的注入风险。文档层面新增并更新了 memory 下的运行手册与状态说明，目的是把“怎么配、怎么跑、失败后怎么看”变成项目内可直接执行的知识，而不是散落在聊天记录里。
